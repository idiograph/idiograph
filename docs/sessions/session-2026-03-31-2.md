## What Was Built

### AMD-014: Port Typing and Type Registry

New Pydantic models:
- `PortType` — registry entry defining a named type: label and description
- `PortDeclaration` — named port slot on a node: name and port_type string reference

New optional fields (non-breaking — all 44 tests unchanged):
- `Node.input_ports: list[PortDeclaration] | None`
- `Node.output_ports: list[PortDeclaration] | None`
- `Edge.from_port: str | None`
- `Edge.to_port: str | None`
- `Graph.type_registry: dict[str, PortType] | None`

## Key Decisions

| # | Decision | Rationale |
|---|---|---|
| 1 | Houdini/Katana port model | Nodes declare typed named slots; edges connect named ports. Production-proven at scale. |
| 2 | port_type is a string reference, not enforced FK | Type registry exists in schema; referential integrity is post-Phase 8. Honest about current state. |
| 3 | None vs empty list semantically distinct | None = ports not typed. Empty list = node explicitly accepts no inputs. Documented in field descriptions. |
| 4 | Mock handlers in arxiv domain module, not core | Domain-specific stubs. Handler registry (AMD-007) makes this clean — executor never imports handlers directly. |
